## Modularity Conventions
- Supabase clients live in `src/lib/supabase/`: `server.ts` (user-context, cookie-bound, RLS applies), `admin.ts` (service_role, server-only, no cookie), `middleware.ts` (proxy-context, used by `src/proxy.ts`). Never construct a Supabase client inline in a route/component — use these factories.
- API route handlers under `src/app/api/` keep validation + response logic inline (no service layer abstracted yet). Frontend must NOT call Supabase directly — go through `/api/*`.
- API auth gate: `src/proxy.ts` protects all `/api/*` by default (fail-closed) with an explicit public allowlist. New protected routes need no extra wiring; new PUBLIC routes must be added to the allowlist in `src/proxy.ts`.
- Page route protection (also `src/proxy.ts`): unauthenticated visits to protected pages redirect to `/login`; logged-in visits to auth pages (`/login`, `/register`) redirect home. New protected/auth pages need TWO updates in `src/proxy.ts`: the runtime constant (`PROTECTED_PAGES`/`AUTH_PAGES`) AND the `config.matcher` static array — Next.js analyzes `matcher` at build time, so a variable/spread reference there is silently ignored. New API routes only need the allowlist; the `/api/:path*` matcher entry already covers them.
- Client-side login-state detection lives in `src/components/AuthNav.tsx` (calls `GET /api/profile` on mount) — not in individual page components. Check there first when a spec assumes a page component owns this logic.
- Shared frontend helpers: `src/lib/auth-client.ts` (fetch wrapper for `/api/auth/*`), `src/lib/validation.ts` (client-side field validation mirroring backend rules), `src/lib/resend-cooldown.ts` (localStorage-backed cooldown timestamp helper, pattern to reuse for any future "cooldown/rate-limited action" UI).
- Payments: all payment-provider logic goes through the `PaymentProvider` adapter in `src/lib/points/provider.ts` (`createCheckout` / `verifyWebhook`) — never call a payment gateway directly from a route handler. Adding a real gateway (e.g. ECPay) means a new adapter implementation only; the checkout/point-granting flow must not change.
- Points: balances are derived (`SUM(delta)` over `point_transactions`, append-only) — never add a mutable balance column or issue UPDATEs to the ledger. Idempotency is owned by the DB (`ref_id` unique + `on conflict do nothing`), not application-level dedup. Ledger/order writes go through `admin.ts` (service_role) only; `authenticated` role has select-own RLS and no insert grant.

## Security Rules (All Agents)
- Never hardcode secrets, tokens, or credentials
- Use environment variables for all sensitive config
- Auth approach is decided: Supabase Auth wrapped by our own API routes; session in httpOnly cookies via `@supabase/ssr`. Flag any auth-adjacent code changes immediately.
- Never log tokens/session/cookies/passwords. Keep anti-enumeration behavior on auth routes (same status + generic message regardless of account existence) — check **status code equality explicitly**, not just message text: a prior bug (register, task 2) leaked account existence via a differing status code (201 vs 200) even though the body text matched. Applies to register/login/resend and any future account-existence-adjacent endpoint.
- `service_role` key (`admin.ts`) is server-only — never import it into client components or expose to the browser.
- Public webhooks (`/api/points/webhook/*`, on the `src/proxy.ts` allowlist) have NO cookie auth — signature verification (HMAC, constant-time compare) is the sole gate. Any new webhook route must verify signatures before touching the DB and rely on DB-level idempotency (`ref_id` unique) for replay safety. Mock provider must refuse to run in production without an explicit `MOCK_PAYMENT_SECRET` (see `getPaymentProvider()`) — never weaken this guard.
